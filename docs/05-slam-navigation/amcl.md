# AMCL 定位

> 地图已经有了，现在要回答：我在哪？

## 第一性原理

AMCL（Adaptive Monte Carlo Localization，自适应蒙特卡洛定位）本质上是：

> “在地图上做很多个机器人假设，然后用激光雷达去淘汰不合理的假设。”

每一个粒子都代表 “机器人可能在这里”。

- 粒子和当前激光扫描匹配得好 → 权重大
- 匹配得差 → 权重小

经过不断更新后，粒子会集中到真正的位置附近。

---

## 在 abot 中的定位角色

启动文件：

- `src/robot_slam/launch/include/amcl.launch.xml`

输入：

- `/scan_filtered`
- `/map`
- `/odom`

输出：

- `map → odom` TF
- `/amcl_pose`
- `/particlecloud`

这里最关键的是：

> **AMCL 不直接告诉电机怎么走，它只是告诉系统 “机器人现在在地图上的哪个位置”。**

而 `move_base` 的全局规划和局部规划，都是建立在这个定位结果之上的。

---

## omni 运动模型意味着什么

当前配置：

```xml
<param name="odom_model_type" value="omni"/>
```

这说明 AMCL 假设你的机器人是**全向底盘**，允许横移，而不是传统差速车。

对于 abot 这类麦克纳姆轮小车，这是合理的。

如果你把 `omni` 误配成 `diff`（差速），会出现两个问题：

1. AMCL 对运动的不确定性建模错误
2. 横向移动时粒子传播方向会错

---

## 5 个 odom_alpha 参数

AMCL 用这 5 个参数描述里程计噪声：

| 参数 | 当前值 | 含义 |
|------|--------|------|
| `odom_alpha1` | 0.2 | 旋转带来的旋转误差 |
| `odom_alpha2` | 0.2 | 平移带来的旋转误差 |
| `odom_alpha3` | 0.2 | 平移带来的平移误差 |
| `odom_alpha4` | 0.2 | 旋转带来的平移误差 |
| `odom_alpha5` | 0.1 | 全向运动额外误差 |

### 如何理解？

这些参数越大，AMCL 就越不信任里程计。

- **里程计很准** → alpha 可以调小
- **轮子容易打滑** → alpha 应该调大

一个非常实用的理解方式是：

> alpha 不是“误差修复参数”，而是“我有多不相信轮子”的参数。

### 调参经验

- 如果机器人走得挺准，但定位粒子散得很开 → alpha 可能太大
- 如果机器人一打滑，定位就完全跑飞 → alpha 可能太小

## 粒子数：`min_particles` / `max_particles`

当前配置：

```xml
min_particles = 500
max_particles = 2000
```

### 含义

AMCL 是“自适应”的：

- 环境明确、定位稳定时，用较少粒子（接近 min）
- 环境复杂、定位不确定时，用更多粒子（接近 max）

### 影响

- 粒子少：速度快，但容易定位不稳
- 粒子多：更稳，但 CPU 占用更高

对比赛小场地来说：

- `500~2000` 是偏保守且稳定的配置
- 如果电脑性能一般，这个范围通常还能接受

## 激光模型参数

### `laser_model_type = likelihood_field`

当前使用的是：

```xml
<param name="laser_model_type" value="likelihood_field"/>
```

这是一种效率较高、工程上常用的激光模型。它不逐束精确建模，而是看当前扫描和地图障碍边界 “有多接近”。

适合 abot 这种室内竞赛小车。

### 关键权重

| 参数 | 当前值 | 含义 |
|------|--------|------|
| `laser_z_hit` | 0.5 | 匹配成功的权重 |
| `laser_z_rand` | 0.5 | 随机噪声的权重 |
| `laser_sigma_hit` | 0.2 | 匹配高斯分布宽度 |

这里一个很有意思的点是：

- `z_hit = 0.5`
- `z_rand = 0.5`

表示系统对 “测量是有意义的” 与 “测量里有随机噪声” 几乎同等看待。这是一种比较保守的设置。

> [!TIP]
>
> 如果赛场障碍物边缘清晰、雷达稳定，可以尝试提高 `laser_z_hit`、降低 `laser_z_rand`，让 AMCL 更相信激光匹配结果。

## 更新阈值

| 参数 | 当前值 | 含义 |
|------|--------|------|
| `update_min_d` | 0.25 | 至少移动 25cm 才更新一次 |
| `update_min_a` | 0.2 | 至少转动 0.2 rad（约 11.5°）才更新一次 |

如果这个值太大，定位更新就会“迟钝”；太小则计算频繁。

对比赛小车来说：

- 0.25m 是偏稳妥的值
- 如果你发现机器人在低速细调时定位更新不及时，可以适当减小到 0.1~0.15

---

## 最重要的问题：恢复机制被禁用

当前配置：

```xml
<param name="recovery_alpha_slow" value="0.0"/>
<param name="recovery_alpha_fast" value="0.0"/>
```

> [!WARNING]
>
> 这意味着 AMCL 的 “自恢复机制” 实际上被关闭了。

### 这有多严重？

当机器人被人搬动、碰撞、轮子打滑严重、或者突然定位跳掉时：

- 正常情况下，AMCL 会尝试重新把粒子撒开，然后重新收敛
- 但现在 recovery 两个 alpha 都是 0
- 结果就是：**定位一旦丢了，就很难自己回来**

### 建议

比赛前建议至少把：

```xml
recovery_alpha_fast = 0.1
```

让 AMCL 在检测到定位异常时有重新发散粒子的能力。

## 如何判断 AMCL 状态是否健康

### 1. 看 `/particlecloud`

在 RViz 中显示 `/particlecloud`：

- 粒子集中在机器人附近 → 健康
- 粒子满地图乱飞 → 定位不确定
- 粒子突然跳到别处 → 发生定位丢失

### 2. 看 `map → odom` TF 是否平稳

- 正常：缓慢变化
- 不正常：突然大跳、来回抖动

### 3. 看机器人在地图中的激光点云

- 正常：扫描轮廓与墙壁重合
- 不正常：扫描整体偏移一截

## 比赛中的 AMCL 使用建议

### 起跑前

- 一定要手动设置初始位姿（2D Pose Estimate）
- 不要指望 AMCL 自己从全图恢复到准确位置

### 比赛中

- 如果发现定位轻微漂，先不要急着重启
- 看粒子云是否还能收回来
- 如果彻底错位，重新打初始位姿比瞎跑更稳

### 调参优先级

如果你要调 AMCL，我建议优先顺序是：

1. `recovery_alpha_fast`
2. `odom_alpha1~5`
3. `min_particles/max_particles`
4. `update_min_d / update_min_a`
