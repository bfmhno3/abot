# TF 坐标变换系统

> 让“我在哪、传感器在哪、目标在哪”进入同一套空间语言

## 为什么 ROS 里一定要有 TF？

机器人不是只有一个坐标系。

真实系统里，至少会同时存在这些参考系：

- 地图坐标系：机器人在整张地图上的位置
- 里程计坐标系：机器人相对起点的连续运动估计
- 机器人本体坐标系：底盘中心在哪里、朝向哪里
- 传感器坐标系：雷达、IMU、摄像头分别装在机器人什么位置
- 目标坐标系：例如 AR 标签、障碍物、目标点

如果没有一套统一的变换机制，你就会不断遇到这种问题：

- 雷达看见的障碍物在 `laser_link` 里，但导航要在 `map` 里规划
- IMU 给出的朝向相对 `imu` 坐标系，但底盘控制关心的是 `base_link`
- 摄像头看见的目标在 `camera_link` 下，但任务节点要判断机器人相对地图的位置

TF 的作用就是把这些原本分散的坐标关系，组织成一套可查询、可组合、可随时间变化的变换系统。

你可以把它理解成：

> TF 是 ROS 里专门管理“空间关系”的公共语言层。

---

## 先建立一个空间层心智模型

理解 TF 最好的方法，不是先背命令，而是先把它分成三个问题：

### 1. 这个数据最初是在哪个坐标系里测出来的？

例如：

- 激光雷达数据天然在 `laser_link`
- IMU 数据天然在 `imu`
- 摄像头识别结果天然在 `camera_link`

### 2. 我现在要在哪个坐标系里使用它？

例如：

- 导航通常想在 `map` 或 `odom` 里思考
- 底盘控制通常想在 `base_link` 里思考
- 可视化工具可能想把一切都放到统一全局坐标系里显示

### 3. 这两个坐标系之间的关系是谁提供的？

例如：

- `base_link -> laser_link` 可能来自 URDF
- `odom -> base_link` 可能来自里程计或 EKF
- `map -> odom` 可能来自 AMCL 或 SLAM

TF 就是在回答这第三个问题，并进一步让前两个问题能够被连起来。

---

## 一张总图：TF 在机器人里做什么？

```text
传感器原始数据
  │
  ├── 激光雷达：laser_link
  ├── IMU：imu
  └── 摄像头：camera_link
  │
  ▼
TF 提供这些坐标系之间的关系
  │
  ├── base_link -> laser_link
  ├── base_link -> imu
  ├── base_link -> camera_link
  ├── odom -> base_link
  └── map -> odom
  │
  ▼
上层模块可以把数据统一换到需要的参考系里
  │
  ├── 导航在 map 中规划
  ├── 控制在 base_link 中发速度
  ├── 视觉在 map 中判断目标方位
  └── RViz 在统一空间中显示系统状态
```

## Frame 到底是什么？

在 TF 里，frame 本质上就是一个坐标参考系的名字。

例如：

- `map`
- `odom`
- `base_link`
- `laser_link`
- `camera_link`

这些名字本身不神秘，关键在于：

> 每个 frame 都代表“以谁为原点、以谁为朝向参考”的一套坐标定义。

### 为什么名字这么重要？

因为在 ROS 里，很多模块不是直接靠“你心里知道这是谁”，而是靠 frame name 来建立空间语义。

比如：

- 某个 LaserScan 说自己来自 `laser_link`
- 某个 PoseStamped 说自己在 `map`
- 某个模块想查询 `map -> base_link`

如果 frame name 错了，即使数值本身看起来像对的，整个系统也会得出错误的空间结论。

---

## TF 树为什么是“树”？

在很多移动机器人项目里，TF 常被画成这样：

```text
map
 └── odom
      └── base_link
           ├── laser_link
           ├── imu
           └── camera_link
```

这个结构有两个核心约束：

### 1. 每个子 frame 只能有一个父 frame

例如：

- `base_link` 不能同时挂在 `odom` 和 `map` 下面
- `laser_link` 不能同时说自己属于两个不同父节点

否则系统就不知道该信哪条空间关系。

### 2. 任意两个 frame 之间的关系应该能沿树路径推出来

例如如果系统已知：

- `map -> odom`
- `odom -> base_link`
- `base_link -> laser_link`

那么它就能推导：

- `map -> laser_link`

这正是 TF 的核心价值之一：

> 你不需要手工给出所有 frame 两两之间的变换，只需要保证树结构完整，系统就能自动组合。

## `map`、`odom`、`base_link` 为什么总被一起讲？

在带定位与导航能力的移动机器人里，它们通常构成一条很常见的主链。

> [!NOTE]
> 并不是所有 ROS1 系统都一定包含 `map`。例如某些早期调试、纯里程计或只关心局部运动的系统，只有 `odom -> base_link -> sensors` 也完全正常。

### `base_link`

它通常表示机器人本体中心坐标系。

很多控制逻辑默认都围绕它展开，因为它最贴近“机器人自己”的视角。

例如：

- 前进方向通常以 `base_link` 前方为正方向
- 底盘速度命令经常默认相对 `base_link` 表达

### `odom`

它通常表示里程计连续积分得到的坐标系。

它的特点通常是：

- **连续**：不会突然大跳变
- **会漂移**：时间长了会逐渐和真实世界偏离

所以它很适合表示“机器人相对起点如何平滑运动过来”，但不适合长期担当全局真值。

### `map`

它通常表示全局定位意义下的稳定参考系。

它的特点通常是：

- 更接近“地图世界”的真实位置
- 可以被 AMCL 或 SLAM 持续修正
- 可能相对 `odom` 发生校正变化

所以导航和定位常常更关心 `map`。

### 三者之间最重要的关系

可以这样记：

- `base_link`：机器人自己
- `odom`：机器人连续运动轨迹的参考
- `map`：机器人在全局世界中的定位参考

通常：

- `odom -> base_link` 描述机器人怎么运动过来
- `map -> odom` 描述全局定位怎么修正里程计漂移

这两层合起来，系统才能同时获得：

- 平滑连续的局部运动
- 受全局定位修正的世界位置

> [!TIP]
> 很多人会问“为什么不直接只留 `map -> base_link`？”因为定位修正和局部连续运动是两类不同需求。把 `map` 和 `odom` 分开，正是为了同时满足“全局正确”和“局部平滑”。

## 静态变换和动态变换有什么区别？

这是 TF 学习里的另一个核心分界。

### 静态变换

指那些不会随时间变化的坐标关系。

例如：

- `base_link -> laser_link`
- `base_link -> imu`
- `base_link -> camera_link`

因为传感器通常是固定安装在机器人上的，所以这些外参不会一直变化。

这类变换常常来自：

- URDF
- robot_state_publisher
- static transform publisher

### 动态变换

指那些会随时间变化的坐标关系。

例如：

- `odom -> base_link`
- `map -> odom`
- 某些目标 frame 相对世界的位置

这类变换通常来自运行中的算法或传感器融合模块。

### 为什么一定要区分这两类？

因为它们的来源和职责不同：

- 静态变换描述的是“装配关系”
- 动态变换描述的是“运动状态”

如果你把这两层混了，系统就很容易出现：

- 重复发布
- frame 含义不稳定
- 调试时搞不清是机械安装问题还是状态估计问题

---

## TF 和 topic 有什么区别？

这也是初学者最容易混淆的一点。

### topic 解决的是“传什么数据”

例如：

- `/odom` 传 Odometry
- `/scan_filtered` 传 LaserScan
- `/imu/data_raw` 传 IMU

它更像数据流本身。

### TF 解决的是“这些数据在什么空间关系里成立”

例如：

- 这个雷达数据属于哪个 frame
- 这个姿态结果是相对哪个参考系定义的
- 当前 `base_link` 在 `map` 下的位置和朝向是什么

### 一个很实用的区分方法

如果你问的是：

- “数值是多少？”
- “消息字段是什么？”
- “谁在发这条消息？”

那通常是在看 topic。

如果你问的是：

- “这个东西相对谁？”
- “从这个坐标系换到那个坐标系怎么变？”
- “机器人和目标之间的空间关系是什么？”

那通常是在看 TF。

## 在 abot 里，这条常见导航链可以怎么落地理解？

前面讲的是通用 TF 基础。回到 abot 这个仓库，可以把它当成一个具体例子来看。

结合仓库现有文档，可以先建立这样一条主链：

```text
map
 └── odom
      └── base_link
           ├── laser_link
           ├── imu
           └── camera_link
```

### 这条主链分别是谁在负责？

常见理解方式是：

- `map -> odom`：由 AMCL / gmapping 这类定位或建图模块提供
- `odom -> base_link`：由里程计或 EKF 融合模块提供
- `base_link -> 各传感器`：由 URDF / robot_state_publisher 提供

### 这条链的工程意义是什么？

它把系统分成三类空间信息：

1. **世界级位置修正**：`map -> odom`
2. **机器人运动状态**：`odom -> base_link`
3. **机器人本体到传感器的安装关系**：`base_link -> sensor`

这样导航、感知、控制、可视化模块就能各自取自己最关心的那一段，而不必每个模块都重新发明坐标变换逻辑。

## TF 查询到底在做什么？

很多人第一次看到代码里的 transform lookup，会感觉像魔法。

其实本质很朴素：

> 你在问系统：“请告诉我，A 坐标系相对 B 坐标系现在是什么位置和朝向？”

例如：

- 查询 `map -> base_link`
- 查询 `base_link -> camera_link`
- 查询 `map -> ar_marker_0`

如果 TF 树完整，而且时间上也能对齐，系统就会把这条路径算出来。

### 为什么查询经常会失败？

常见原因通常不是 API 用错，而是以下几类：

- 某段变换根本没人发布
- frame name 写错
- frame 之间不在同一棵树里
- 查询的时间点上没有可用数据
- 节点启动太早，依赖 frame 还没就绪

这也是为什么很多 TF 问题，最后都不是“数学问题”，而是“系统组织问题”。

---

## 时间为什么会影响 TF？

TF 不是只管空间，还隐含管“这个空间关系在什么时刻成立”。

这很重要，因为机器人在动。

例如：

- 1 秒前 `base_link` 在这里
- 现在 `base_link` 已经走到别处

如果你拿当前雷达数据，却配上 1 秒前的机器人姿态，空间解释就会错位。

### 对学习者来说，先记住这件事就够了

> TF 不是单纯的静态几何表，而是“随时间变化的空间关系系统”。

所以当你以后看到：

- extrapolation error
- lookup timeout
- transform unavailable

不要只从空间方向想，也要从时间方向想。

---

## 一个非常重要的误区：TF 不是“谁都可以随便发”

很多初学者会觉得：

- 我知道 `odom -> base_link`
- 你也知道 `odom -> base_link`
- 那我们都发一下也没问题

这是错误的。

同一条关键变换如果有多个发布者，就会产生冲突。

最典型的就是：

- 轮式里程计在发 `odom -> base_link`
- EKF 也在发 `odom -> base_link`

如果系统没有明确谁接管，TF 树就会变得不可信。

### 更稳妥的理解是

- 每条关键变换最好有明确唯一来源
- 如果用了融合模块，就让融合模块成为该变换的权威发布者
- 其他模块改为提供输入，而不是继续抢着发布同一条 TF

---

## 为什么 TF 问题经常表现得像“导航问题”或“视觉问题”？

因为上层模块依赖 TF，但它们报出来的现象往往不是“TF 三个字”。

例如：

- 导航方向怪异，可能不是 DWA 参数错，而是 `base_link` 朝向定义不对
- RViz 显示传感器位置漂移，可能不是传感器坏了，而是 frame 关系错了
- 视觉追踪对不准目标，可能不是 PID 错了，而是 `map -> target` 或 `map -> base_link` 查错了

也就是说：

> TF 经常是“表面问题不写 TF，根因却在 TF”的那一层基础设施。

## 在 abot 里，哪些模块会明显依赖 TF？

前面的内容解决的是“TF 作为通用机制到底在做什么”。如果回到 abot 这个具体项目，可以把下面这些模块当成观察样本，而不是把它们当成 TF 定义本身。

### 1. 导航与定位

- AMCL / gmapping 负责全局位置修正
- EKF / odom 负责机器人运动估计
- move_base 依赖稳定的空间关系来规划与控制

### 2. 姿态滤波

在姿态滤波相关文档里已经能看到，滤波器不仅输出姿态结果，也可能发布与姿态相关的 TF。

### 3. 视觉追踪与 AR 标签

在 `docs/06-perception/ar-tag-tracking.md` 中可以看到一种典型用法：

- 直接监听 `/tf`
- 查询 `map -> ar_marker_*`
- 查询 `map -> base_link`
- 再基于这两个空间关系计算机器人应如何转向目标

这说明在视觉模块里，TF 不只是辅助显示，而是直接参与控制逻辑。

## 学 TF 时最推荐的阅读顺序

如果你现在拿到一个陌生机器人项目，建议按这个顺序理解 TF：

1. **先找常见导航链**：如果项目有全局定位，先看是否存在 `map -> odom -> base_link`
2. **再找传感器挂点**：`base_link -> laser_link / imu / camera_link`
3. **再找关键发布者**：谁在发每一段变换
4. **最后再看谁在查询这些变换**：导航、视觉、控制分别依赖哪一段

这样你会先得到一张完整空间骨架，而不是一上来陷进 API 细节。

---

## 一个学习者最该建立的 TF 认知

1. TF 解决的不是“消息传输”，而是“空间关系统一”。
2. `map`、`odom`、`base_link` 分别承担全局定位、局部连续运动、机器人本体参考三种不同职责。
3. 静态变换描述装配关系，动态变换描述运动状态。
4. TF 问题经常伪装成导航、视觉、控制问题。
5. 学 TF 的关键不是先背工具命令，而是先看清“谁相对谁、谁发给谁、谁在查谁”。
