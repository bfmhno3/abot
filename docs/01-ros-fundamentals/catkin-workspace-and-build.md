# Catkin 工作空间与构建流程

> 从“源码如何组织”到“节点如何被编译、安装、加入 ROS 环境”的主线说明

## 为什么 ROS 需要自己的工作空间工具？

如果你只写一个普通 C++ 程序，CMake 已经足够。

但在 ROS1 里，你面对的不是一个单体工程，而是一组互相依赖的包：

- 一个包定义消息
- 一个包提供驱动
- 一个包做定位
- 一个包做导航
- 上层任务包又常常依赖前面很多内容

这时需要解决的不只是“怎么编译一个工程”，而是：

1. **包和包之间怎么声明依赖**
2. **多个包如何放在同一工作空间下构建**
3. **构建结果如何让其他包继续找到**
4. **运行节点时，系统如何知道可执行文件和库在哪里**

Catkin 就是 ROS1 为这些问题提供的工作空间与构建约定。

---

## 先建立一个构建层心智模型

很多人第一次学 catkin 时，会把它理解成“ROS 自带的某个编译命令”。这不算错，但太浅。

更准确的理解应该是：

> Catkin 不是单个命令，而是一整套围绕“多包工作空间”建立起来的组织方式。

它同时在解决三件事：

1. **包级依赖如何表达**
2. **多个包如何在一个工作空间中被构建**
3. **构建结果如何重新暴露给运行时和后续构建过程**

如果没有这一层约定，你就会在机器人项目里不停手动处理：

- 先编哪个包
- 后编哪个包
- 哪些头文件路径要传给谁
- 哪些库要先装到哪里
- 哪些环境变量要手动设置

当包数一多，这套事情会迅速失控。

---

## 一张总图：从源码到运行

```text
src/ 下的多个 ROS 包
        │
        ▼
读取 package.xml + CMakeLists.txt
        │
        ▼
catkin_make / catkin build 构建工作空间
        │
        ├── build/   构建中间结果
        ├── devel/   开发期可直接使用的结果
        └── install/ 安装期标准化结果（可选）
                │
                ▼
       source setup.bash 加入环境
                │
                ▼
      rosrun / roslaunch 能找到你的节点和包
```

这个流程里最容易被忽略的一点是：

- 构建不是终点
- 构建结果还必须被“注册进环境”

否则你虽然已经把可执行文件和库编出来了，但 ROS 运行时仍然不知道应该去哪里找它们。

## 工作空间目录结构

一个典型 ROS1 工作空间在概念上长这样：

```text
工作空间根目录/
├── src/
│   ├── CMakeLists.txt
│   ├── abot_base/
│   ├── robot_slam/
│   └── ...
├── build/
├── devel/
└── install/
```

### `src/`

源码区。真正需要编辑的包都在这里。

更具体地说，`src/` 里装的不是“一个工程的源码”，而是“多个 ROS 包的源码集合”。

这也是 ROS 学习中非常重要的一次思维切换：

> 工作空间不是单个项目目录，而是多个包共同构成的系统工作区。

### `build/`

构建中间目录。里面是 CMake 配置结果、编译缓存、目标文件等。

你通常不会直接修改这里的东西。它更像是构建过程临时展开的工作台。

### `devel/`

开发空间。构建完成后，很多可执行文件、库、Python 模块、环境脚本会在这里形成一个可直接 `source` 的结果空间。

它的重点不是“长得像最终安装目录”，而是：

- 你刚编完就能用
- 适合开发调试
- 适合反复修改、反复构建、反复验证

### `install/`

安装空间。它更接近标准化发布结果，适合需要更干净部署输出的场景。

你可以把它理解成“更接近正式安装后的布局”。

> [!NOTE]
> 对学习者来说，最重要的理解是：`devel/` 面向开发调试，`install/` 面向更正式的安装式结果。不要把它们理解成“一个对一个错”，而是用途不同。

## `package.xml` 和 `CMakeLists.txt` 分别管什么？

这是学习 catkin 时最容易混淆的一组文件。

### `package.xml`

它是**包元数据与依赖声明**。

常见内容包括：

- 包名
- 版本
- 维护者
- 许可证
- 依赖包

例如：

```xml
<package>
  <name>abot_bringup</name>
  <depend>roscpp</depend>
  <depend>rospy</depend>
  <depend>std_msgs</depend>
</package>
```

你可以把它理解成“这个包是谁、依赖谁”。

### `CMakeLists.txt`

它是**当前包的构建脚本**。

它负责描述：

- 需要找哪些依赖
- 源码该怎么编译
- 头文件路径在哪里
- 目标可执行文件和库怎么生成
- 哪些内容需要安装

你可以把它理解成“这个包具体怎么建”。

> [!WARNING]
> 很多旧笔记会把这两者都概括成“写依赖”。这太粗糙了。更准确的说法是：`package.xml` 声明包级依赖关系，`CMakeLists.txt` 决定当前包的构建过程。

### 为什么 ROS 要两份文件同时存在？

因为它们服务的是不同层次的问题。

#### `package.xml` 解决的是“包级关系”

它告诉工作空间工具：

- 这个包依赖谁
- 谁应该先构建
- 这个包的元信息是什么

也就是说，它更像“包之间的关系说明书”。

#### `CMakeLists.txt` 解决的是“当前包内部怎么建”

它告诉底层构建系统：

- 哪些源文件要编译
- 目标名叫什么
- 头文件目录在哪里
- 要链接哪些库

也就是说，它更像“这个包自己的施工图”。

### 一个很容易记住的比喻

- `package.xml`：这个包对外介绍自己是谁、依赖谁
- `CMakeLists.txt`：这个包对内安排自己怎么被造出来

这个区分一旦清楚，catkin 学习会顺很多。

## 最常见的原生 catkin 命令

### `catkin_init_workspace`

在原生 catkin 工作流里，它会在 `src/` 下准备顶层 `CMakeLists.txt`。

```bash
cd ~/catkin_ws/src
catkin_init_workspace
```

### 为什么会有这个命令？

因为原生 catkin 工作流需要一个顶层入口，让整个工作空间被当作 catkin 工作空间识别并组织起来。

对学习者来说，重点不是记住它到底创建了什么链接，而是知道：

- 它是工作空间初始化的一部分
- 它服务的是原生 catkin 工作流
- 如果你用的是 `catkin_tools`，理解它的存在即可，不必把它当成现代主入口

### `catkin_create_pkg`

快速创建新包骨架。

```bash
catkin_create_pkg my_robot_controller roscpp std_msgs geometry_msgs
```

### 这个命令真正帮你省了什么？

它不是只省了“手敲几个文件”的时间，而是帮你避免：

- 漏掉基础字段
- 依赖声明不完整
- `package.xml` 和 `CMakeLists.txt` 起步结构不规范

所以它更像一个“符合 ROS 规范的脚手架生成器”。

### `catkin_make`

ROS1 教程里最常见的工作空间构建命令。

```bash
cd ~/catkin_ws
catkin_make
source devel/setup.bash
```

它的优势是：

- 入门简单
- 官方教程覆盖广
- 小型工作空间足够用

### 为什么它在教程里这么常见？

因为它是学习曲线最低的入口之一：

- 不需要先理解太多配置
- 命令很直接
- 很多老教程都围绕它写

对初学者来说，这种低门槛很重要。但一旦工作空间变大、包变多，使用体验就不一定是最舒服的了。

## `catkin_make`、`catkin_make_isolated`、`catkin build` 的区别

### 1. `catkin_make`

特点：

- ROS1 教程里最常见
- 入门门槛低
- 适合小型或学习型工作空间
- 整体构建体验偏“统一式”

### 2. `catkin_make_isolated`

特点：

- 也是原生工具链的一部分
- 更强调包级隔离
- 常见于一些特殊构建或兼容场景
- 使用频率通常低于前两者

### 3. `catkin build`（来自 `catkin_tools`）

特点：

- 来自 `catkin_tools`，不是最早的原生默认入口
- 工作空间管理更现代
- 包级隔离、选择性构建、配置管理体验更好
- 对大型工作空间更友好

> [!TIP]
> 更准确的说法是：在 Melodic 时代，很多教程默认从 `catkin_make` 讲起；而 `catkin build` 是更现代、通常也更舒服的工作空间工具。

### 用第一性原理怎么理解这三者？

#### `catkin_make`

你可以把它理解成：

> 更偏“把整个工作空间当成一个整体来构建”。

它适合入门，因为简单；但当包多了以后，它的统一式体验有时会让问题边界不够清晰。

#### `catkin_make_isolated`

你可以把它理解成：

> 原生工具链对“包级隔离构建”需求的一种补充。

它让你更容易把问题定位到单个包层面，但整体使用体验不如后来的 `catkin_tools` 顺手。

#### `catkin build`

你可以把它理解成：

> 更现代的工作空间管理入口，更强调包级操作、配置管理和选择性构建。

所以很多实际项目里，一旦工作空间变大，大家会更偏向它。

### 学习阶段应该怎么选？

- 你在读老教程：常会碰到 `catkin_make`
- 你在维护稍大的工作空间：更常会喜欢 `catkin build`
- 你在看一些特殊兼容场景：可能会见到 `catkin_make_isolated`

重点不是“谁绝对先进”，而是知道它们各自解决什么使用场景。

如果你的工作流使用 `catkin_tools`，最常见的是这些：

### `catkin build`

```bash
catkin build
```

作用：构建整个工作空间，或构建指定包。

它的真正价值在于：

- 你可以按包思考问题
- 你可以按包选择性构建
- 你不必每次都把整个工作空间重新当作黑盒处理

### `catkin config`

```bash
catkin config --install
catkin config -DCMAKE_BUILD_TYPE=Release
```

作用：保存工作空间级配置。

这说明 `catkin_tools` 不只是“执行一次构建命令”，而是在管理一个有状态的工作空间配置。

### `catkin clean`

```bash
catkin clean
catkin clean some_pkg
```

作用：清理构建结果，既可以全局清理，也可以只清某个包。

它在排查缓存、错误产物、错误配置遗留时特别有用。

### `catkin profile`

```bash
catkin profile add debug
catkin profile set debug
```

作用：在同一工作空间管理多套构建配置。

你可以把它理解成“同一个源码区，对应多种构建宇宙”。

### `catkin locate`

```bash
catkin locate -s some_pkg
catkin locate -b
```

作用：快速定位源码目录或构建目录。

它的价值虽然不如 build/config 那么核心，但在工作空间变大后会非常顺手。

## `devel` 和 `install` 应该怎么理解？

### `devel`

它面向开发阶段：

- 构建完成后可直接 `source devel/setup.bash`
- 适合边改边编、边编边测
- 重点是“快速进入可运行状态”

### `install`

它面向更干净、更标准化的结果布局：

- 用于更正式的部署或打包思路
- 更接近标准安装后的目录组织

如果使用 `catkin_tools`，更常见的写法是：

```bash
catkin config --install
catkin build
```

而不是把 `install` 当作普通 make 参数去理解。

> [!NOTE]
> 旧笔记里常把 `devel` 简化成“全是符号链接伪装出来的虚拟 FHS”，这说法太绝对。更稳妥的理解是：`devel` 提供一个开发期可直接使用的结果空间，具体落地形式取决于工具和配置。

### 为什么 ROS 需要这两个空间？

因为开发和发布是两种不同诉求。

#### `devel` 的诉求是“快”

你刚改完代码，希望立刻：

- 重新构建
- `source`
- 启动验证

所以它优先服务开发效率。

#### `install` 的诉求是“稳”和“规整”

当你更关心：

- 标准化目录布局
- 更干净的部署结果
- 打包与分发思路

这时 `install` 更符合目标。

### 学习者容易犯的错

#### 错误 1：把 `devel` 理解成“假的 install”

这会让你一直在追究它底层是链接还是复制，而忽略它真正的角色：

- 它首先是开发期结果空间
- 不是“装成 install 的某种骗局”

#### 错误 2：把 `install` 理解成“高级版 devel”

也不准确。

它们不是高低级关系，而是用途差异。

## FHS 为什么只需要在这里讲一小节？

FHS（Filesystem Hierarchy Standard）本来是 Linux 世界对安装布局的约定。

你只需要抓住它和 ROS 学习最相关的部分：

- `include/`：头文件
- `lib/`：库与很多包内可执行目标
- `share/`：launch、URDF、消息定义、配置等架构无关资源

ROS 之所以关心这些目录，是因为它也需要把自己的构建结果组织成“别人能找到”的形式。

但对于 ROS 初学者来说，FHS 不是主线知识。真正主线是：

1. 工作空间怎么组织
2. 包依赖怎么声明
3. 构建命令怎么用
4. 环境为什么要 `source setup.bash`

所以 FHS 在这里作为辅助理解就够了，不需要单独变成 01 章的主文档。

### 为什么很多旧笔记会把 FHS 讲得特别重？

因为一旦开始追问“构建结果最后到底放到哪里”，很容易一路深入到底层目录布局。

这本身没有错，但对入门者来说，过早把重心放在 FHS 上，会让主线跑偏。

更好的顺序是：

- 先理解工作空间和构建流程
- 再理解结果空间为什么需要有目录约定
- 最后再把 FHS 当成辅助背景知识接进来

## 为什么经常需要重新 `source setup.bash`？

因为构建结果虽然已经生成，但当前 shell 并不会自动知道：

- 你的包在哪里
- 新生成的节点在哪里
- 新生成的 Python 模块在哪里
- CMake 再次找包时应该先去哪里找

所以常见步骤是：

```bash
source devel/setup.bash
```

或在 install 模式下：

```bash
source install/setup.bash
```

你可以把它理解成：

> “把这个工作空间注册进当前终端环境。”

### 为什么这一步这么关键？

在新开终端、切换 overlay，或者你怀疑当前环境还没更新时，重新 `source` 往往是最稳妥的默认动作。

因为构建生成的是文件，而不是自动修改当前终端上下文。

如果不 `source`，你很可能遇到这些现象：

- `rosrun` 找不到刚编出来的节点
- `roslaunch` 找不到你的包
- Python 找不到新生成的模块
- CMake 下一次找依赖时没走你以为的路径

于是就会出现一种很典型的“明明编出来了却像没编一样”的错觉。

这类问题在 ROS 学习早期特别常见。

---

## 一个学习者最该建立的构建认知

1. 很多 ROS 构建错误来自依赖声明、构建脚本或环境没有接好，不一定是编译器本身的问题。
2. 工作空间是包的集合，不是单个项目目录。
3. `catkin_make` / `catkin build` 解决的是“能不能生成结果”，`rosrun` / `roslaunch` 解决的是“生成结果后能不能运行起来”。
4. `package.xml`、`CMakeLists.txt`、`setup.bash` 分别负责依赖关系、当前包构建过程和把结果接进当前环境。
