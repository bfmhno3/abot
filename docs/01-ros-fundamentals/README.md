# ROS1 第一性原理

> 从“为什么需要 ROS”开始，先建立对节点、通信、坐标系、启动方式与构建工具的整体认知。

## 为什么这一章要先学？

如果你直接跳进 `abot_driver`、`move_base` 或 `mission_node` 的源码，很容易陷入“每个文件都认识一点，但不知道整套系统为什么这样组织”的状态。

这一章的目标不是背 API，而是先建立三层认知：

1. **ROS 在解决什么问题**：为什么机器人系统需要统一的通信框架和工具链。
2. **系统是怎么被拆开的**：节点、话题、服务、动作、TF 各自承担什么职责。
3. **代码是怎么被启动和构建的**：launch 文件怎么组织系统，catkin 工作空间怎么把多个包编译起来。

当这三层认知建立之后，你再去读后面的驱动、导航、任务代码，就不容易把“运行问题”“通信问题”“构建问题”混在一起。

---

## 一张总图：从源码到机器人行为

```text
源码目录中的多个 ROS 包
        │
        ▼
catkin 工作空间完成构建与环境组织
        │
        ▼
launch 文件把节点、参数、命名空间拼装起来
        │
        ▼
多个 ROS 节点开始运行
        │
        ├── 通过 topic 持续交换数据
        ├── 通过 service 发起请求/响应
        ├── 通过 action 管理长时间任务
        └── 通过 TF 统一各坐标系关系
                │
                ▼
        最终形成 abot 的整机行为
```

## ROS 在解决什么问题？

造机器人时，你面对的通常不是一个单体程序，而是一组必须协同工作的模块：

- 底盘驱动要和上层导航通信
- IMU、雷达、摄像头要持续发布数据
- 定位与建图模块要共享坐标变换
- 任务逻辑要在不同阶段调度不同能力

如果这些模块都靠手写 socket、手动维护进程、各自定义数据格式，系统会很快失控。

ROS（Robot Operating System）解决的核心问题是：**为机器人软件提供统一的通信模型、工具链和工程组织方式。**

> [!NOTE]
> ROS 不是传统意义上的“操作系统内核”。更准确地说，它是一组围绕机器人开发组织起来的库、工具和约定。

## 核心概念速览

### 节点（Node）

节点是独立运行的程序单元。一个节点通常负责一个明确职责，例如：驱动底盘、读取 IMU、执行导航、管理比赛任务。

你可以把它理解成：

> 一个 ROS 系统，本质上是一组彼此协作的进程。

### 话题（Topic）

话题是**发布 / 订阅**通信。适合持续流动的数据：

- `/odom`
- `/cmd_vel`
- `/scan_filtered`
- `/imu/data_raw`

它强调的是“持续广播、松耦合接收”。

### 服务（Service）

服务是**请求 / 响应**通信。适合一次性动作或查询，例如触发 IMU 校准。

### 动作（Action）

动作适合**耗时较长、需要反馈、允许取消**的任务，例如导航到某个目标点。

### TF

TF 用来描述各坐标系之间的空间关系，例如：

- `map`
- `odom`
- `base_link`
- `laser_link`
- `camera_link`

如果没有 TF，多传感器数据就很难在同一个空间语义下协作。

> [!TIP]
> 初学 ROS 时，最重要的不是死记定义，而是建立“数据如何流动、坐标如何统一、节点如何协作”的整体图景。

## 学这一章时最容易混淆的几件事

### 1. 不要把“运行问题”和“构建问题”混为一谈

- `catkin_make` / `catkin build` 解决的是：能不能把包构建出来
- `roslaunch` / `rosrun` 解决的是：构建出来之后怎么启动
- `rostopic` / `rosnode` / `rosparam` 解决的是：启动之后系统是否按预期运行

这三类问题的排查入口完全不同。

### 2. 不要把 ROS 系统想成一个单体程序

更贴近现实的理解是：

- 每个节点做一部分事
- 节点之间通过标准接口通信
- launch 文件负责拼装系统
- catkin 工作空间负责组织和构建源码

### 3. 不要把 launch、CLI、catkin 看成同一层东西

- **CLI** 偏运行与调试
- **launch** 偏系统启动与配置
- **catkin** 偏源码组织与构建

把它们分层理解，后面看仓库时会清晰很多。

## 学习这一章时的建议

如果你目前还没有 ROS1 运行环境，也完全可以先把这章当成“阅读源码的字典”。

建议带着三个问题去看：

1. 这个节点为什么要单独存在？
2. 这个数据为什么走 topic，而不是 service？
3. 这个功能属于“运行时问题”“启动配置问题”还是“构建问题”？

只要能持续把新知识归到这三个问题下面，你对整个 abot 仓库的理解就会越来越稳。
